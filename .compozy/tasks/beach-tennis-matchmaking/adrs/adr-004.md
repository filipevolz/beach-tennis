# ADR-004: JWT Authentication with Single-Role RBAC

## Status

Accepted

## Date

2026-05-16

## Context

The marketplace has two actor types: players and venue managers. Endpoints must enforce who can publish availability, create matches, join spots, and confirm bookings. The MVP needs stateless API authentication suitable for a separate Next.js frontend and future clients.

## Decision

Implement **JWT-based authentication** with short-lived **access tokens** and **refresh tokens** (stored httpOnly cookie on web for refresh, or body for mobile clients later).

**RBAC roles** (enum): `player`, `venue_manager`.

**MVP rule**: each user account has **exactly one role** assigned at registration. Authorization guards on NestJS controllers check role + resource ownership (e.g., venue manager can only mutate their venue's slots).

Password auth for MVP (email + password, bcrypt). OAuth providers deferred to Phase 2.

## Alternatives Considered

### Alternative 1: Multi-Role per Account from MVP

- **Description**: One login can be both player and venue manager.
- **Pros**: Fewer accounts for owner-operators who also play.
- **Cons**: More complex UI routing, permission matrix, and testing surface in MVP.
- **Why rejected**: User chose single role per account for MVP; document path to multi-role in Phase 2 without blocking launch.

### Alternative 2: Session Cookies Only (No JWT)

- **Description**: Server-side sessions with cookie session ID.
- **Pros**: Easy revocation, no token leakage in JS.
- **Cons**: Sticky sessions or shared session store for scale; less natural for future native apps.
- **Why rejected**: JWT fits API-first separation; refresh token in httpOnly cookie mitigates XSS for renewal.

### Alternative 3: Managed Auth (Clerk, Auth0, Supabase Auth)

- **Description**: Outsource identity to a SaaS provider.
- **Pros**: Faster social login, less security maintenance.
- **Cons**: Vendor cost, coupling, custom claims for venue ownership still needed in app DB.
- **Why rejected**: MVP scope is email/password; add provider when OAuth is prioritized.

## Consequences

### Positive

- Clear guard boundaries per module (`@Roles('venue_manager')`).
- Stateless access tokens simplify horizontal API scaling.

### Negative

- Token revocation requires short access TTL + refresh rotation strategy.
- Users who are both player and venue need two accounts until Phase 2.

### Risks

- Role confusion in UI if registration flow is unclear.
- **Mitigation**: Explicit role selection at signup; no role switching in MVP settings.

## Implementation Notes

- NestJS: `AuthModule`, `JwtStrategy`, `RolesGuard`, `@CurrentUser()` decorator.
- JWT payload: `sub` (user id), `role`, `email`.
- User table links to `PlayerProfile` or `VenueManagerProfile` based on role.
