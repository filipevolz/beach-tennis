# ADR-003: PostgreSQL with Prisma and PostGIS

## Status

Accepted

## Date

2026-05-16

## Context

The MVP requires geospatial discovery (nearby matches and venue slots), relational integrity between venues, court availability, open matches, spots, and bookings, and predictable schema evolution as the marketplace grows. The backend is NestJS with no existing ORM choice.

## Decision

Use **PostgreSQL** as the primary database with the **PostGIS** extension for location queries (distance, bounding box). Use **Prisma** as the ORM and migration tool. Store coordinates as `geography(Point, 4326)` (or Prisma `Unsupported` + raw SQL where needed) with indexed spatial queries for discovery filters.

Run Postgres locally via Docker Compose for development; target managed Postgres (e.g., RDS, Neon, Supabase Postgres) for production.

## Alternatives Considered

### Alternative 1: PostgreSQL + TypeORM

- **Description**: TypeORM entities with NestJS integration.
- **Pros**: Common in NestJS tutorials, decorator-based entities.
- **Cons**: Migration and relation ergonomics weaker than Prisma for small teams; PostGIS typing less smooth.
- **Why rejected**: User chose Prisma; Prisma migrations and schema clarity favor MVP speed.

### Alternative 2: SQLite for MVP, Postgres Later

- **Description**: File-based DB for local MVP, migrate later.
- **Pros**: Zero infra for first demos.
- **Cons**: No PostGIS parity, migration pain, false confidence before production.
- **Why rejected**: Geo discovery is core to PRD; Postgres from day one avoids rework.

### Alternative 3: MongoDB

- **Description**: Document store for flexible profiles and matches.
- **Pros**: Flexible schema for evolving match metadata.
- **Cons**: Weaker relational guarantees for booking ↔ slot ↔ match; geo queries less standardized than PostGIS.
- **Why rejected**: Booking and slot inventory need transactional consistency and joins.

## Consequences

### Positive

- ACID transactions for booking confirmation (prevent double-booking a slot).
- PostGIS supports "near me" discovery efficiently.
- Prisma schema is a single source of truth for models and migrations.

### Negative

- PostGIS adds operational requirement (extension enabled on host).
- Prisma raw queries needed for some spatial operations until Prisma native geo matures.

### Risks

- N+1 queries in discovery lists.
- **Mitigation**: Paginate discovery endpoints; use selective `include` in Prisma; add indexes on `(status, starts_at)` and spatial columns.

## Implementation Notes

- Schema: `backend/prisma/schema.prisma`.
- NestJS: `PrismaService` as global provider; repositories as thin wrappers per module if testing requires mocks.
- Seed script for dev: sample venues, slots, players, public matches.
