# ADR-002: NestJS Backend with Next.js Frontend

## Status

Accepted

## Date

2026-05-16

## Context

The repository is greenfield with empty `backend/` and `frontend/` directories. The MVP requires a two-sided marketplace API, role-based access for players and venue managers, and a mobile-first web experience. A single-language TypeScript stack reduces context switching and enables shared types between client and server.

## Decision

Use **NestJS** for the REST JSON API in `backend/` and **Next.js (App Router)** with React in `frontend/`. Organize the repo as a **pnpm workspace** monorepo with a shared types package only when duplication becomes painful (not in MVP initial scaffold).

Communication: **REST over HTTPS** with JSON request/response bodies. OpenAPI document generated from NestJS decorators for frontend client generation (optional in MVP, recommended before Phase 2).

## Alternatives Considered

### Alternative 1: Go Backend + React Frontend

- **Description**: Go services with explicit interfaces, separate React SPA.
- **Pros**: Strong performance, clear interface boundaries, aligns with some Compozy skill examples.
- **Cons**: No type sharing with frontend, two language ecosystems, slower MVP iteration for a small team.
- **Why rejected**: User preference for NestJS + React; TypeScript end-to-end is a better fit for rapid MVP delivery.

### Alternative 2: TypeScript Full Stack on Next.js Only (Route Handlers)

- **Description**: Single Next.js app with API routes and no separate backend service.
- **Pros**: Fewer deployables, simplest repo layout.
- **Cons**: Harder to enforce domain boundaries, venue-manager and player APIs grow entangled, weaker fit for future mobile native clients.
- **Why rejected**: Marketplace domains (venue supply, match demand, booking) benefit from a dedicated API service with NestJS modules.

### Alternative 3: Python FastAPI + React

- **Description**: FastAPI backend with Pydantic models and React frontend.
- **Pros**: Fast prototyping, good OpenAPI support.
- **Cons**: Split language stack, less alignment with Next.js/React hiring and ecosystem for this product.
- **Why rejected**: Not aligned with chosen stack.

## Consequences

### Positive

- Shared TypeScript knowledge across backend and frontend.
- NestJS modules map cleanly to PRD bounded contexts (player, venue, match, booking).
- Next.js supports SSR/SSG where useful and PWA for mobile-first MVP.

### Negative

- NestJS boilerplate heavier than minimal frameworks for very small APIs.
- Two deployables (API + web) vs. a single Next.js app.

### Risks

- Over-abstraction in NestJS modules early on.
- **Mitigation**: Start with flat module structure per domain; extract shared libraries only when duplicated.

## Implementation Notes

- Entry: `backend/src/main.ts`, modules under `backend/src/`.
- Frontend: `frontend/app/` for routes, `frontend/features/` for domain UI.
- CORS configured for frontend origin; API base URL via environment variable.
