# ADR-005: Venue Slots Only; Public and Invite-Only Matches

## Status

Accepted

## Date

2026-05-16

## Context

The PRD open questions asked whether venues publish only availability or also host matches, and whether the MVP supports private organization. Confusion between "booking a court" and "joining a match" is a stated product risk. Technical entities must stay distinct while supporting both public discovery and small-group organization.

## Decision

**Venue supply (MVP)**: Venues publish **availability slots** only (`AvailabilitySlot`) — court + time window + capacity. Venues do **not** create `Match` records in MVP.

**Player demand**: Players create **open matches** (`Match`) for singles or doubles with skill level and format. Matches list **spots** (`MatchSpot`) with participation status.

**Visibility**:
- `visibility: public` — appears in discovery feeds (filtered by geo, level, format).
- `visibility: invite` — hidden from public discovery; join via unguessable `inviteCode` (URL-safe token).

**Booking**: When a match is ready to lock a venue time, the organizer (or designated player) creates a **Booking** linking `matchId` + `availabilitySlotId`, transitioning slot to reserved on confirmation.

## Alternatives Considered

### Alternative 1: Venues Create Hosted Matches

- **Description**: Arena publishes "open match" events players join directly.
- **Pros**: Venues drive demand on quiet slots.
- **Cons**: Two match creation paths, heavier venue UX, conflicts with ADR-001 player-centric participation loop.
- **Why rejected**: User chose slots-only for venues; keeps venue flow lightweight per PRD.

### Alternative 2: Public Discovery Only

- **Description**: All matches visible in browse; no private flows.
- **Pros**: Simplest discovery index and security model.
- **Cons**: Ignores real-world groups that already have 2–3 players and need one more.
- **Why rejected**: User wants invite-link matches without listing them publicly.

### Alternative 3: Merge Slot and Match into One Entity

- **Description**: Single "Game" entity for both venue inventory and player gatherings.
- **Pros**: Fewer tables.
- **Cons**: Blurs booking vs. join semantics; harder to enforce venue occupancy rules.
- **Why rejected**: PRD explicitly mitigates user confusion by separating concepts.

## Consequences

### Positive

- Clear API resources: `/availability-slots`, `/matches`, `/bookings`.
- Invite flow is a column + index, not a separate product surface.
- Venue MVP stays minimal (profile + publish slots).

### Negative

- Players must understand two-step flow: form a match, then book a slot.
- Invite matches need share UX (copy link) in frontend.

### Risks

- Orphan matches never booked.
- **Mitigation**: Match status `forming` → `ready_to_book` → `booked`; optional expiry job in Phase 2.

## Implementation Notes

- `Match.inviteCode` nullable, unique when `visibility = invite`.
- Discovery queries always filter `visibility = public` AND `status IN ('forming', 'open')`.
- Booking endpoint validates slot is `available` and match has required spots filled (configurable rule: allow booking while still filling last spot if organizer confirms).
