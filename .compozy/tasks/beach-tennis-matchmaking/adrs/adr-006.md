# ADR-006: Mobile-First Web via Next.js PWA

## Status

Accepted

## Date

2026-05-16

## Context

The PRD states users make real-time decisions on mobile and the experience must be easy to scan. Native apps are out of scope for MVP timeline. A installable, responsive web app reduces store friction while supporting launch in a limited geography.

## Decision

Deliver the player and venue manager UI as **Next.js (App Router)** with responsive layouts and **PWA** capabilities: web app manifest, service worker for asset caching (not offline-first data in MVP), and add-to-home-screen support.

Defer React Native until Phase 3 unless retention metrics justify earlier investment.

## Alternatives Considered

### Alternative 1: Vite React SPA Without PWA

- **Description**: Client-only SPA, no service worker.
- **Pros**: Simpler build, no SW caching pitfalls.
- **Cons**: Weaker mobile "app-like" presence, no install prompt.
- **Why rejected**: User chose Next.js PWA for mobile-first positioning.

### Alternative 2: React Native from MVP

- **Description**: Native iOS/Android apps first.
- **Pros**: Best mobile UX, push notifications native.
- **Cons**: Two platform pipelines, slower iteration, app store delay for launch.
- **Why rejected**: Over-scoped for MVP; web PWA validates demand first.

## Consequences

### Positive

- Single codebase for venue and player web flows.
- PWA improves re-engagement without app store.

### Negative

- iOS PWA limitations (push, background) vs. native.
- Service worker adds deployment/cache-bust discipline.

### Risks

- Performance on low-end devices if pages are heavy.
- **Mitigation**: Server components for lists, minimal client JS on discovery routes, image optimization.

## Implementation Notes

- Use `next-pwa` or equivalent; cache static assets only in MVP.
- Touch-friendly UI: large tap targets, sticky CTAs on match detail.
- Geolocation: browser API for "near me" with fallback to manual city/region (launch geography configurable via env).
